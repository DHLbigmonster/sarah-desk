## 这东西到底能干什么？

Sarah 是一个驻留在 macOS 菜单栏的语音助手。按一下热键开始说话，再按一下结束，它就把你说的话变成文字，或者帮你干活。

三种模式，一个热键搞定：

| 模式 | 热键 | 效果 |
|------|------|------|
| 听写 | 按一下触发键（默认 Right Ctrl） | 你说的话直接打到光标位置，任何 App 都行 |
| 命令 | 触发键 + Shift | 你说的话交给 AI Agent 去执行——打开网页、搜索、写飞书文档 |
| 快问 | 触发键 + 空格 | 你说的话变成一个问题，AI 在浮窗里直接回答你 |

热键还能自定义。Right Ctrl、Right Alt、CapsLock、F1–F12，随便你选。在 Settings → Hotkeys 里改就行。

**【配图建议 2】**：三种模式的示意图或对比图。可以用三张小图拼在一起——左边是听写模式下文字出现在编辑器里，中间是命令模式下 Agent 在操作，右边是快问模式下浮窗回答问题。让读者一眼看懂三种模式的区别。

## 最有价值的三个功能

### 第一：按一下说话，再按一下就打字

这个体验一旦用过就回不去了。

以前用语音输入，你得打开某个 App，找到输入框，点一下麦克风图标，等它加载，说话，等识别，确认……摩擦太大了。

Sarah 不一样。你在任何 App 里——VS Code、微信、飞书、浏览器——按一下 Right Ctrl，说话，再按一下。文字直接出现在光标位置。不用切窗口，不用点按钮，不用等。

而且它不是简单地把语音转成文字就完事了。后面还有一层 Refinement：去口头禅、补标点、重组逻辑。短句快速清洗，长句智能结构化。你嘴里说出来的是口语，打到屏幕上的像写过的文章。

**【配图建议 3】**：听写模式的实际使用场景。比如在 VS Code 里按一下热键说话，代码注释直接出现的 GIF 或截图。这张图要让读者感受到"无缝"。

---

### 第二：说话就能让 AI 帮你干活

Command 模式是我最得意的部分。

按一下触发键 + Shift，说一句"帮我搜一下 React 19 的新特性"，再按一下结束。Sarah 会先截一张你当前屏幕的图——对，在它自己的窗口弹出来之前就截了——然后把截图、当前 App 名称、窗口标题、URL 全部打包交给 AI Agent。

Agent 看到的是你正在看的东西，不是它自己的界面。

然后它就开始干活了。打开浏览器搜索、读取页面内容、把结果整理好呈现给你。全程你不用碰鼠标。

更厉害的是，Agent 还能接入飞书。说一句"帮我在飞书文档里新建一个会议纪要"，它直接调用飞书 API 去创建。说一句"把这段话发到项目群里"，它就发了。

**【配图建议 4】**：Command 模式的完整流程截图。建议用一组图展示：(1) 用户在浏览器里看一个页面 (2) 按热键说话 (3) Agent 覆盖层出现，显示它正在执行任务 (4) 最终结果展示。如果是 GIF 动图更好。

### 第三：零配置就能用

这是我觉得最重要的设计决策。

很多语音助手工具，上来就让你注册账号、申请 API Key、填一堆配置。对开发者来说不算什么，但对普通用户来说就是一道墙。

Sarah 的语音识别有两套后端。主后端是火山引擎的流式 ASR，识别精度高、响应快。但如果用户没有配置火山引擎的 Key——没关系，自动切换到 Apple Speech，macOS 自带的语音识别，零配置，离线也能用。

你下载下来，打开，授权麦克风权限，就能开始说话了。就这样。

**【配图建议 5】**：首次启动引导流程的截图。展示 Sarah 弹出的权限申请通知，以及设置界面中"已连接"的状态。让读者知道"开箱即用"不是说说而已。

## 技术上做了什么

不展开太多细节，挑几个有意思的说。

**五个窗口，各司其职。** Sarah 不是一个窗口走天下。它有五个独立的 Electron 窗口：一个隐藏的录音窗口（负责 Web Audio 采集）、一个浮动 HUD 胶囊（显示语音状态和波形）、一个 Agent 覆盖层（显示 AI 执行结果）、一个迷你设置面板（菜单栏弹出的控制中心）、一个完整的调试控制台。每个窗口有自己的 Vite 构建配置，互不干扰。

**上下文感知。** Command 模式下，Sarah 会在自己的窗口弹出之前截取用户当前屏幕。这意味着 Agent 看到的是你正在看的网页、正在编辑的文档、正在看的代码——而不是 Sarah 自己的界面。这个时序细节看起来小，但对 Agent 的执行质量影响巨大。

**凭证加密存储。** 所有 API Key 用 Electron 的 safeStorage 加密后存在本地，不用明文写 .env 文件。设置界面里直接编辑，改完立即生效。

**【配图建议 6】**：五个窗口的拆解图。可以用一张大图展示 Sarah 的界面组成——菜单栏图标、浮动 HUD、Agent 覆盖层、迷你设置面板、调试控制台，标注每个部分的功能。

## 调试控制台：不止是个语音助手

Sarah 还内置了一个完整的调试控制台（ClawDesk），侧边栏导航，能管理 Agents、Skills、Models、Sessions，能直接聊天、设置定时任务、配置 MCP 服务器。

Settings 里能改的东西更多：通用设置、热键配置、API Provider、CLI 路径、Skills 管理——全部可视化，不用碰命令行。

## 这个项目是怎么做出来的

全程用 Claude Code 开发。不是"AI 辅助写几个函数"那种用法，是真正把 Claude Code 当成主力开发工具。

从架构设计到代码实现，从 bug 修复到 UI 打磨，从 CI/CD 配置到开源文件准备——几乎所有代码都是在 Claude Code 里完成的。

有几个经验想分享：

**1. 给 AI 足够的上下文。** 我在项目里维护了一个详细的 CLAUDE.md，记录了架构决策、语音模式行为、热键配置、ASR 后端、凭证存储、Agent 任务队列、上下文捕获逻辑、签名打包策略——所有 AI 需要知道的信息。有了这个文件，AI 的输出质量直接上一个台阶。

**2. 一次只做一件事。** 不要贪多。先让录音能跑起来，再加语音识别，再加文字清洗，再加 Agent 模式，再加 UI 打磨。每一步都测通了再进下一步。这个节奏比"一口气全做完"快得多。

**3. 不要怕推翻重来。** 中间有一次我发现 intent-router 的设计太复杂了，直接删掉，改成三个模式统一走 OpenClaw。代码少了，逻辑清晰了，bug 也少了。AI 最大的优势就是不怕重写，你怕什么？

**【配图建议 8】**：可选——开发过程中的截图或对话记录。比如 Claude Code 在终端里输出的开发日志，或者 Git 提交历史的截图。让读者感受到"一天做完"的密度。
